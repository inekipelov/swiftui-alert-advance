# Planning Role

## Overview

Use this guide when the task is ambiguous, exploratory, design-oriented, or explicitly about planning before implementation.

## Rules

### Brainstorming

- Use the [$brainstorming] skill for planning tasks, new features, behavior changes, and any work that needs design clarification before implementation.
- Treat `brainstorming` as a hard gate before implementation work: explore context, clarify intent, compare approaches, present a design, and get approval before moving forward.
- For planning-heavy requests, keep the conversation in design mode until the user explicitly approves the next implementation step.

### Execution Boundary

- Do not edit code while the task is still in planning, scoping, or design clarification.
- If the user asks for analysis, strategy, trade-offs, or a plan, answer in planning mode rather than jumping to implementation.
- Once the user gives a clear implementation instruction, switch to the relevant task-specific guide and keep the change set small.
